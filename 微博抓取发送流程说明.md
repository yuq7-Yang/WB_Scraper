# 微博抓取与回复流程说明

本文说明当前微博美甲美睫采集系统的整体逻辑，包括：数据怎么抓、抓到哪里、页面怎么显示、真实回复怎么发送、微博返回结果怎么处理。

## 1. 总体流程

```text
页面按钮
  ↓
Flask 接口 /api/scrape 或 /api/reply
  ↓
后台线程执行任务
  ↓
抓取模块或回复模块
  ↓
微博接口返回 JSON
  ↓
更新 SQLite 数据库状态
  ↓
页面自动刷新显示：待回复 / 已预演 / 已回复 / 失败
```

## 2. 抓取入口

页面点击“开始采集”后，会请求：

```text
POST /api/scrape
```

这个接口在：

```text
weibo_bot/dashboard.py
```

它接收页面传来的参数：

```text
keywords            关键词
max_per_keyword     每个关键词最多采几条
max_total           总共最多采几条
```

然后后台调用：

```text
weibo_bot/scraper.py
run_scrape()
```

## 3. 抓取具体步骤

`run_scrape()` 会做这些事：

```text
1. 检查 Scrapfly Key 有没有配置
2. 检查微博 Cookie 有没有配置
3. 初始化 SQLite 数据库
4. 遍历关键词
5. 用关键词搜索微博帖子
6. 拿到帖子 ID
7. 进入每个帖子抓评论
8. 筛选华东地区评论
9. 保存到 SQLite
```

## 4. 搜索帖子接口

搜索微博帖子使用的是微博移动端搜索接口：

```text
https://m.weibo.cn/api/container/getIndex
```

代码位置：

```text
weibo_bot/scraper.py
search_posts()
```

程序会把关键词拼进搜索接口里，然后从返回结果里提取帖子 ID。

## 5. 抓取评论接口

拿到帖子 ID 后，程序会请求这个评论接口：

```text
https://m.weibo.cn/comments/hotflow?id=帖子ID&mid=帖子ID&max_id_type=0
```

代码位置：

```text
weibo_bot/scraper.py
fetch_comments()
```

这个接口返回帖子下面的评论列表。

## 6. Scrapfly 的作用

程序不是直接硬冲微博，而是通过 Scrapfly 去请求微博接口。

可以理解为：

```text
你的程序 → Scrapfly → 微博
```

Scrapfly 的作用是帮程序代访问微博接口，并带上微博 Cookie。

## 7. 地区筛选逻辑

抓到评论后，程序会看评论里的来源字段：

```text
source
```

例如：

```text
来自上海
来自浙江
来自江苏
来自福建
```

只要属于华东地区，就会保存。

判断函数在：

```text
weibo_bot/scraper.py
is_east_china()
```

## 8. 数据保存在哪里

数据保存在当前项目目录下的 SQLite 数据库：

```text
weibo.db
```

数据库相关代码在：

```text
weibo_bot/db.py
```

主要字段如下：

```text
user_name       评论用户昵称
location        来源地区
comment_text    用户评论内容
comment_id      这条评论自己的微博 ID
post_id         原帖 ID
keyword         是哪个关键词抓到的
status          当前状态
reply_text      你回复的话术
replied_at      回复时间
```

## 9. post_id 和 comment_id 的区别

这是整个系统里最重要的两个 ID。

```text
post_id     原帖 ID
comment_id  某一条评论自己的 ID
```

五岁版理解：

```text
post_id 是教室编号
comment_id 是某个小朋友的座位号
```

只知道 `post_id`，只能对整个帖子发表评论。

知道 `comment_id`，才能回复到某个人的评论下面。

## 10. 页面怎么显示数据

页面会定时请求：

```text
GET /api/leads
```

接口位置：

```text
weibo_bot/dashboard.py
api_leads()
```

它会从 SQLite 里读取所有线索，然后返回给页面。

页面根据数据库里的 `status` 显示状态：

```text
pending   待回复
reviewed  已预演
replied   已回复
failed    失败
```

所以页面看到的“已预演”“已回复”，本质上来自数据库的 `status` 字段。

## 11. 发送入口

页面选择话术后点击“确认发送”，会请求：

```text
POST /api/reply
```

接口位置：

```text
weibo_bot/dashboard.py
api_reply()
```

页面会传这些数据：

```json
{
  "lead_ids": [27],
  "reply_text": "11",
  "confirm_real_send": true
}
```

其中最关键的是：

```text
confirm_real_send
```

它表示页面上有没有勾选“我确认真实发送评论”。

## 12. 预演和真实发送的区别

回复逻辑在：

```text
weibo_bot/replier.py
reply_to_lead()
```

核心判断是：

```python
if ENABLE_REAL_REPLIES and confirm_real_send:
    真发微博
else:
    只做预演
```

必须同时满足两个条件，才会真实发送：

```text
1. .env 里面 ENABLE_REAL_REPLIES=true
2. 页面里勾选“我确认真实发送评论”
```

如果没有勾选确认框，系统不会真的发微博，只会把数据库状态改成：

```text
reviewed
```

页面显示：

```text
已预演
```

如果真实发送成功，数据库状态会改成：

```text
replied
```

页面显示：

```text
已回复
```

## 13. 真实发送前先拿 st 令牌

真实发送函数在：

```text
weibo_bot/replier.py
_send_real_reply()
```

它会先请求：

```text
https://m.weibo.cn/api/config
```

目的是确认：

```text
Cookie 是否仍然登录
有没有 st 令牌
```

`st` 可以理解成微博给你的临时通行证。

没有 `st`，就不能发评论。

## 14. 回复具体评论

如果数据库里有 `comment_id`，程序会调用：

```text
https://m.weibo.cn/api/comments/reply
```

发送参数是：

```python
{
    "content": "11",
    "mid": "原帖ID",
    "cid": "目标评论ID",
    "st": "微博通行证"
}
```

这里最关键的是：

```text
cid
```

`cid` 就是目标评论 ID。

有了它，微博才知道你是在回复某一条评论，而不是评论整篇帖子。

## 15. 评论原帖

如果数据库里没有 `comment_id`，程序只能调用：

```text
https://m.weibo.cn/api/comments/create
```

发送参数是：

```python
{
    "content": "11",
    "mid": "原帖ID",
    "st": "微博通行证"
}
```

这个接口的效果是：

```text
评论整篇帖子
```

不是回复某个人的评论。

## 16. reply 和 create 的区别

```text
api/comments/reply   回复某条评论下面
api/comments/create  评论整个帖子
```

所以：

```text
有 comment_id    → 用 reply
没有 comment_id  → 用 create
```

## 17. 微博返回怎么判断成功

微博接口会返回 JSON。

程序主要看这个字段：

```text
ok
```

代码逻辑是：

```python
if int(data.get("ok", 0)) <= 0:
    raise RuntimeError(...)
```

意思是：

```text
ok > 0   认为微博接受成功
ok <= 0  认为失败
```

## 18. 成功后的数据库状态

如果微博返回成功，程序会更新数据库：

```text
status = replied
reply_text = 你选择的话术
replied_at = 当前时间
```

页面刷新后显示：

```text
已回复
```

## 19. 失败后的数据库状态

如果微博返回失败，或者 Cookie 失效，或者接口报错，程序会更新数据库：

```text
status = failed
reply_text = 错误信息
```

页面刷新后显示：

```text
失败
```

## 20. 页面怎么收到任务进度

发送或采集任务开始后，接口会先返回：

```json
{"status": "started"}
```

这句话的意思是：

```text
后台任务已经开始
```

不是说任务已经完成。

真正的采集和发送在后台线程里继续执行。

后台线程会通过：

```text
/stream
```

把进度推给页面。

页面收到后会刷新表格，重新请求：

```text
GET /api/leads
```

再显示最新状态。

## 21. 当前系统的完整链路

```text
采集：
页面 /api/scrape
  → run_scrape()
  → 搜索微博帖子
  → 抓帖子评论
  → 筛选华东地区
  → 保存到 SQLite

显示：
页面 /api/leads
  → 读取 SQLite
  → 显示待回复 / 已预演 / 已回复 / 失败

发送：
页面 /api/reply
  → run_reply()
  → reply_to_lead()
  → 判断是否真实发送
  → 获取微博 st
  → 有 comment_id 就回复评论
  → 没有 comment_id 就评论原帖
  → 根据微博返回 ok 更新数据库
```

## 22. 一句话总结

```text
抓取时：用 Scrapfly + 微博 Cookie 找帖子和评论，存进 SQLite。
发送时：用微博 Cookie + st 令牌调用微博评论接口。
返回时：看微博 JSON 里的 ok，成功就标记已回复，失败就标记失败。
页面显示：只读 SQLite 里的 status。
```

## 23. 后续可以增强的地方

目前真实发送成功后，程序主要保存：

```text
status
reply_text
replied_at
```

以后可以增加一个字段：

```text
reply_comment_id
```

这样每次真实回复成功后，都能保存“我发出去的那条回复自己的微博 ID”。

这样以后可以更精确地追踪、核对、撤回或复查每条回复。
